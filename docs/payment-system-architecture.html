MM Payment System — Architecture at a glance

Branch: vs/mm-payment-mvp

Merchant and customer share the same MetaMask Mobile app. EVM-only v1, EIP-681 URIs with extra merchant metadata query params. All entry points are gated behind the local MM_PAYMENT_SYSTEM_ENABLED env flag.

Merchant entry

New "Request" action on the Wallet home, next to Buy/Swap/Send/Receive.

Customer entry

QR scanner or ethereum: deeplink — both route merchant URIs to PayMerchantConfirmation.

Core utility

app/util/payment-request — build, parse, route, atomic-amount helpers.

Feature flag

process.env.MM_PAYMENT_SYSTEM_ENABLED === 'true' gates navigation, entry button, and routing helper.

Out of scope (v1)

SoftPOS / card payments

Feasibility only — see docs/payment-system-card-feasibility.md. Non-code blockers (compliance, KYC, settlement) make it unviable for v1.

Non-EVM chains

EIP-681 is EVM-specific. Solana / Bitcoin / etc. would need a separate URI scheme and asset-selection flow.

Merchant-side receipt

Merchant verifies payment through normal wallet activity; no passive poller or push receipt yet.

Fiat amount toggle

Form takes crypto amount + decimals. Fiat input / toggle is a follow-up.

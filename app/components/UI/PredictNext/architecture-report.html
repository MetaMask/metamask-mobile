§ Executive Summary

The Predict feature shipped fast under product uncertainty, accumulating a 2,600+ line god controller, a 3,100+ line god provider, 96 hooks, and 56 component directories. PredictNext is a ground-up redesign around deep modules with slim interfaces (Ousterhout): six focused services replace the monolith, a single canonical VenueAdapter contract bounds venue complexity, and a 3-tier UI taxonomy collapses variant explosion. Migration runs inside-out over 7 phases — internals are replaced first, UI second — so every PR is shippable.

Key bets

  • Deep services, slim controller. PredictController shrinks to a stateless composition root with two methods (initialize, destroy). Each service registers as a first-class Engine.context entry and owns its own Redux slice via BaseController/BaseDataService.
  • One venue contract. A single VenueAdapter interface bounds all protocol translation. PredictClient is a derived session-bound view of it, produced exclusively by PredictSessionService. No separate client interface to drift.
  • Errors defined out of existence. Services absorb retries, reconnects, optimistic patches, and venue-specific failures. UI sees only four categories: empty_state · unavailable · action_failed · degraded.
  • Direct cache coordination. Cache-mutating cross-service calls (e.g. TradingService → PortfolioService.onOrderSubmitted) are constructor-injected method calls, not Service Events. Service Events are observation-only.
  • Inside-out migration. Legacy PolymarketProvider and old controller progressively delegate to new internals while the old UI stays unchanged. UI migrates as vertical slices only after data layer is proven in production.

1 Current Architecture (Legacy Predict/)

The legacy feature lives at app/components/UI/Predict/. Two god objects dominate the architecture and pull complexity upward into hooks and components. The data model uses legacy terminology (Market / Outcome / OutcomeToken) that no longer matches venue or product language.

  • 51 public methods on one class
  • Owns order state machine, reads, transactions, subscriptions, analytics, eligibility, geo
  • Extends BaseController; state shape mixes activeBuyOrders, payment token, etc.

  • 30+ public methods mixing concerns
  • HTTP: fetchEventsFromPolymarketApi, getOrderBook, getBalance
  • TX building: buildProtocolUnsignedOrder, buildDepositMaintenanceTransaction, buildClaimTransaction
  • Caching: GameCache, TeamsCache, WebSocketManager
  • Signing & auth: signProtocolOrder, createApiKey
  • Transform: parsePolymarketEvents, parsePolymarketPositions

Layering smells (concrete locations)

PolymarketProvider is a god object
providers/polymarket/PolymarketProvider.ts · 3,133 LoC
Mixes HTTP, transaction building, caching, transformation, signing, optimistic patches, and protocol-specific logic in one file. Lines 100–148 import HTTP utilities and TX builders. placeOrder() (≈177 lines) does preview + TX build + sign + submit. Any change ripples across unrelated concerns.
Hooks bypass the controller for reads
hooks/ · 75 direct Redux interactions
Many hooks call useSelector directly on the Predict slice instead of going through messenger actions. This makes data flow ambiguous and tests brittle.
Variant duplication in components
components/ · 56 directories
5 skeleton variants, 4+ card variants, 4+ button variants, 4+ outcome display variants. The same product concept reappears with slight visual changes instead of a single compound primitive with variants.
Business logic inside hooks
usePredictPlaceOrder.ts (311 LoC), usePredictToastRegistrations.tsx (450+ LoC)
Order placement, deposit chaining, and toast plumbing live in hooks rather than services. Hooks become the orchestration layer, defeating the controller boundary.
Custom caches outside any data-service abstraction
providers/polymarket/GameCache.ts, TeamsCache.ts, WebSocketManager.ts
Bespoke caches with their own TTL and invalidation logic, parallel to React Query. No shared dedup / retry / circuit-breaker policy.

2 Target Architecture (PredictNext/)

PredictNext follows Ousterhout's A Philosophy of Software Design: modules should be deep (large complexity hidden behind a small public API). The result is fewer public methods, fewer cross-layer dependencies, and fewer states UI must understand.

Four-layer stack

Layer 4 · Components
Views compose Widgets compose Primitives. Primitives are compound (Vercel-style). 7 primitives replace 56 directories.
Layer 3 · Hooks
Domain folders with barrel exports. Data hooks are granular (one query each). Imperative hooks (useTrading, useTransactions, useLiveData) stay deep.
Layer 2 · Services + Composition Root
6 services + 1 helper. State-owners extend BaseController; read services extend BaseDataService. PredictController only bootstraps the graph.
Layer 1 · Venue Adapters
Single canonical VenueAdapter contract. PolymarketAdapter + future KalshiAdapter are stateless protocol translators.

Service inventory

ModuleBase classPublic surfaceOwns / hides
PredictController Plain (composition root) 2 methods: initialize, destroy Service instantiation, transactional fail-closed bootstrap, feature lifecycle. No Redux state.
PredictSessionService BaseController 3 actions + state slice Session lifecycle, signer resolution, Account Readiness ownership, eligibility, invalidation. Produces session-bound PredictClient view.
MarketDataService BaseDataService 8 actions Events / search / carousel / price-history / prices. Stale-time policy. Cache patches from live updates.
PortfolioService BaseDataService 5 actions + cache-coord methods Positions / activity / balance / PnL. Optimistic patches via onOrderSubmitted / Confirmed / Failed.
TradingService BaseController 5 actions + state machine slice Order state machine, rate limiting, funding-via-private-executor, direct cache notifications to PortfolioService.
TransactionService Plain (stateless) 3 public actions + private executor Public deposit / withdraw / claim; private TransactionExecutor used by TradingService for order funding.
LiveDataService Plain (stateless) 2 actions + status Socket lifecycle, reconnection, multiplexing. Calls MarketDataService.applyPriceUpdates / PortfolioService.applyPortfolioUpdate directly.
predictAnalytics Injected helper one track() method Not a service. Constructed in composition root, injected by constructor reference into emitting services.

3 Migration Plan — 7 Phases

Foundation

PRs: 2–3 Deps: None Risk: Freeze gate required

Establish canonical domain types, VenueAdapter contract, PredictSessionService contract, error model, and bidirectional translation layer.

  • Canonical types: PredictEvent, PredictMarket, PredictOutcome, PredictPosition, …
  • VenueAdapter interface + derived PredictClient type alias

Polymarket Adapter & PredictSessionService

PRs: 4–5 Deps: Phase 1 Risk: Characterization tests required

Stateless PolymarketAdapter + PredictSessionService (BaseController, produces session-bound PredictClient). Legacy PolymarketProvider progressively delegates method by method.

  • Polymarket venue API clients: gammaClient, clobClient, dataApiClient, liveDataClient
  • DTO + mapper modules
  • Acceptance: adapter stays stateless, session service owns lifecycle, legacy provider returns identical shapes, no changes to old hooks/controllers

Read Services

PRs: 3–4 Deps: Phase 2 Parallel with: Phase 4

MarketDataService + PortfolioService on BaseDataService. Old controller delegates reads.

  • Cache strategy: events 5 min · prices 1 min · positions 1 min · balance ~1 min
  • Service integration tests with mocked PredictClient
  • Open: cache invalidation semantics under-specified — needs PR-level definition

Write Services

PRs: 4–5 Deps: Phase 2 Parallel with: Phase 3

TradingService (state machine), TransactionService (public actions + private executor), LiveDataService, predictAnalytics helper.

  • TradingService → PortfolioService: direct constructor refs for onOrderSubmitted / Confirmed / Failed
  • TradingService → TransactionExecutor: private executor for order funding (no user-intent analytics)
  • LiveDataService → {MarketData,Portfolio}Service.apply*: direct cache-coord calls
  • Open: exact method signatures for live cache-coord need clearer specification
5

PredictController Composition Root

PRs: 1–2 Deps: Phases 3, 4 Risk: Transactional fail-closed bootstrap

New PredictController exposes only initialize / destroy. Constructs predictAnalytics first, then services in the correct order, with reverse-order teardown on any construction failure.

  • Six services as first-class Engine.context entries; helper not registered
  • Old controller becomes a thin shim for legacy callers
  • Open: boot-blocking vs boot-degrading failure-mode mapping should appear in error-handling.md too
6

UI Migration (Vertical Slices)

PRs: 8–12 Deps: Phase 5

One screen at a time: hooks + components + view + component view tests per slice. No screen mixes old and new hooks.

  • Granularity rule: each data hook triggers exactly one query / subscription
  • Three-tier UI: Primitives (pure) · Widgets (wire data) · Views (compose)
  • Order: event feed (home) → event details → portfolio → order flow → modals → external embeds

Cleanup

Delete old Predict/, delete compat/, git mv PredictNext/ → Predict/, update remaining external imports.

  • Hard gate: zero runtime imports from old Predict/
  • No "PredictNext" terminology remains in production code or import paths

5 Risks & Open Questions

The architecture is internally consistent and well-aligned with the migration plan. The remaining risks are about enforcement gaps, under-specified semantics, and a few structural choices that might still be improvable. Those are explored in the architectural-improvement pass that follows this report.

Enforcement gaps

  • Query keys are owned by the interface ledger but not validated at compile time. A service can use a new key shape without anyone noticing.
  • Public exports are listed in the ledger but index.ts is not auto-checked against it.
  • Hook names follow a naming rule (e.g. useEventDetail, not useEvent) but there is no linter to enforce it.
  • Error registry is a runtime object; using a code not in the registry silently falls back to UNKNOWN.
  • Stale-time strategy is documented but each service author can pick differently.

Under-specified semantics

  • Cache invalidation (Phase 3): which Service Events or direct calls trigger invalidation, at what granularity, with what rollback on failure?
  • Live cache-coord method signatures (Phase 4): exact shapes of applyPriceUpdates / applyPortfolioUpdate, what to do with updates for uncached keys, how to handle out-of-order updates.
  • Fail-closed bootstrap failure modes (Phase 5): which init failures are boot-blocking vs boot-degrading should appear in both services.md and error-handling.md.
  • Component view test coverage (Phase 6): minimum % coverage, required interaction types, snapshot policy.
  • Cache fallback "safe": error-handling.md mentions returning cached data when "safe" but never defines safety thresholds.

Structural questions worth re-examining

  • Wide VenueAdapter interface (~30 methods). Could be split by capability domain — but doing so before Kalshi exists might be premature.
  • Implicit cache-coordination graph: direct method calls between services replace Service Events, but the system-of-record graph is not written down anywhere. A small "cache-coord map" would surface coupling.
  • Compound component context escape: EventCard sub-components can be imported and used outside their parent context. Type-level enforcement is possible.
  • Imperative hook state leakage: useTrading returns both the Redux-backed activeOrder and imperative placeOrder / preview / cancel. Splitting reads from writes is worth considering.
  • Plain "stateless service" naming: TransactionService and LiveDataService are described as "plain services" but register on the Engine context like real controllers. The split with the analytics "helper module" is asymmetric.
